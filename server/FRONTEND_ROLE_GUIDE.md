# 🛡️ دليل التعامل مع نظام الأدوار (Role Management Guide)

هذا الدليل مخصص لمطوري الواجهة الأمامية (Frontend) لفهم كيفية التعامل مع نظام الأدوار المؤمن في الـ Backend.

## 1. فلسفة النظام (Internal vs Public)

يعتمد النظام على فصل كامل بين ما يتم تخزينه "سرياً" وبين ما يظهر للمستخدم "علنياً":
- **المفاتيح السرية (Secret Keys)**: مخزنة في ملف الـ `.env` وقاعدة البيانات فقط (مثل: `66de7be5...`). لا تصل أبداً للمتصفح.
- **الأدوار العامة (Public Roles)**: هي مسميات بسيطة يتعامل معها الفرونت إند (مثل: `admin`, `artisan`, `buyer`).

## 2. كيفية التعامل مع الأدوار في الفرونت إند

### أ- عند التسجيل (Registration)
عند إرسال طلب إنشاء حساب جديد، يجب إرسال الدور ككلمة نصية بسيطة:
```json
{
  "name": "Sami",
  "email": "sami@example.com",
  "role": "artisan" // الخيارات المتاحة: 'artisan' أو 'buyer' فقط
}
```
**ملاحظة**: لا يمكن للفرونت إند إنشاء حساب `admin`. السيرفر سيرفض الطلب فوراً.

### ب- عند جلب البيانات (Profile / Me)
عند استدعاء مسار `/api/auth/me` أو `/api/admin/users` ستصلك البيانات بالشكل التالي:
```json
{
  "id": "...",
  "name": "...",
  "role": "buyer" // ستصلك الكلمة البسيطة دائماً
}
```

### ج- التحكم في الواجهة (UI Logic)
يمكنك استخدام الدور القادم من السيرفر لإظهار أو إخفاء عناصر معينة في الواجهة:
```javascript
// مثال في React أو Vue
const userRole = user.role;

if (userRole === 'admin') {
  return <AdminDashboard />;
} else if (userRole === 'artisan') {
  return <ArtisanProfile />;
}
```

## 3. لماذا هذا النظام آمن؟
1. **منع التلاعب (IDOR Protection)**: حتى لو عرف المستخدم كلمة `admin` وحاول إرسالها، السيرفر يتحقق من الصلاحيات (Authorization) بناءً على التوكن المشفر وليس على الكلمة المرسلة.
2. **حماية المعلومات (Information Hiding)**: المهاجم لا يعرف القيم الحقيقية للأدوار في قاعدة البيانات، مما يجعل هجمات الحقن (Injection) أو التخمين شبه مستحيلة.
3. **سهولة الصيانة**: إذا قررنا تغيير المفتاح السري في السيرفر لأسباب أمنية، لن يحتاج مطور الفرونت إند لتغيير سطر واحد في كوده.

---
**تنبيه**: لا تقم أبداً بتخزين أي "مفاتيح سرية" داخل كود الفرونت إند. اعتمد دائماً على المسميات العامة التي يوفرها الـ API.
